# ZenPulse (AI Meditation App)

Мобильное приложение на **React Native + Expo** для медитаций и генерации персональных аффирмаций.

## Что делает приложение

- показывает список медитаций (часть контента может быть premium);
- даёт выбрать текущее настроение;
- генерирует персональную аффирмацию под выбранный mood;
- содержит paywall-экран и базовую логику подписки;
- поддерживает сохранение состояния подписки локально.

## Технологии

- React Native 0.81
- Expo SDK 54
- TypeScript
- React Navigation (native stack)
- AsyncStorage

## Запуск проекта

1. Установить зависимости:

```bash
npm install
```

2. Создать `.env` на основе `.env.example` и указать ключ OpenRouter:

```env
EXPO_PUBLIC_OPENROUTER_API_KEY=your_openrouter_api_key
```

3. Запустить dev-сервер Expo:

```bash
npx expo start
```

4. Открыть приложение:

- `a` для Android-эмулятора;
- `i` для iOS-симулятора (на macOS);
- либо через Expo Go по QR-коду.

## Генерация аффирмаций через OpenRouter API

Генерация текста выполняется в `src/ai/affirmations.ts` через HTTP-запрос к:

- `https://openrouter.ai/api/v1/chat/completions`

Используется модель:

- `arcee-ai/trinity-large-preview:free`

Если API-ключ не задан, либо сеть/ответ недоступны, приложение автоматически переключается на mock-генерацию (локальная библиотека фраз), чтобы UI не ломался и сценарий оставался рабочим.

## Структура (кратко)

- `src/screens` — экраны (`Meditations`, `Affirmation`, `Paywall`);
- `src/components` — переиспользуемые компоненты (карточки, mood selector);
- `src/ui` — UI-примитивы (кнопки, badge, glass card);
- `src/state` — локальное состояние подписки;
- `src/theme` — токены цветов, отступов, типографики.

## Ответ на вопрос

**«С какими специфическими проблемами мобильной верстки ИИ справляется хуже всего и как ты контролировал его работу, чтобы приложение не сломалось на маленьких экранах (iPhone SE vs Pro Max)?»**

Хуже всего ИИ обычно справляется с:

- адаптацией длинных текстов и бейджей в узкой шапке (конфликты `title + badge + action`);
- предсказанием реальной высоты контента при разных шрифтах/локалях;
- горизонтальными каруселями (неверный `snap`, слишком широкие карточки);
- «магическими» фиксированными размерами, которые выглядят хорошо только на одном устройстве.

Как контролировалось в этом проекте:

- задавались ограничители ширины контента (`width: "88%"`, `maxWidth`, `alignSelf: "center"`);
- использовались `SafeAreaView` и явные вертикальные отступы между секциями;
- для конфликтных зон в шапке применялись `flex: 1`, `minWidth: 0`, `flexShrink`, `gap`;
- убирались хрупкие `className`-зависимости в пользу явного `StyleSheet`;
- размеры текста и карточек уменьшались до стабильных значений на узких экранах;
- проверялись граничные сценарии: короткий/длинный контент, разные mood-эмодзи, premium-бейдж.

Это позволило держать макет устойчивым между маленькими и большими экранами без «разъезда» блоков.
